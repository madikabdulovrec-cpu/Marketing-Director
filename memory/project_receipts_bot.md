# Receipts Telegram Bot (cheki-bot)

Telegram-бот для сбора чеков от кассиров на 17 АЗС (@ts_cheks_bot).

**GitHub:** [Evimiz-KZ/cheki-bot](https://github.com/Evimiz-KZ/cheki-bot) — **отдельный** репозиторий, не зеркалится в Marketing-Director (см. `.gitignore`).

## Функционал

- Регистрация кассира: Фамилия → Имя → № АЗС (реальные номера из STATION_NUMBERS, 17 шт.)
- Кассир выбирает акцию кнопкой (Lipton / Зурай / Прочая), шлёт фото → бот сохраняет с тегом
- **Единица счёта — акционный ТОВАР** (решение Мадияра 2026-07-03): обычный чек = 1 товар;
  комбо-чек после валидации = количество товаров, заданное админом по каждой акции
  (`receipt_promotions.qty`). Все рейтинги АЗС/групп, разбивки и итоги кампаний — в товарах.
- **Комбо-чеки (2+ товара)**: кнопка «🎁 2+ товара» → фото в очередь валидации
  (`review_status='pending'`), бот спрашивает у кассира число товаров (кнопки 1–10,
  подсказка админу, не блокер). Админ в «🔔 Валидация» степперами − / + задаёт количество
  по каждой акции (до 10) и утверждает, либо отклоняет с причиной (файл удаляется).
- **Multi-tag хранение**: junction `receipt_promotions(receipt_id, promotion_code, qty)` /
  `receipt_campaigns` — источник истины. `/retag <id>` — перекатегоризация с количествами.
  Approved «без акций» = 0 товаров; legacy-чеки без тегов = 1.
- **Кампании**: жизненный цикл акций (старт → сбор → закрытие с summary_json). Партиальное
  закрытие: multi-tag чек выживает, если у него остались другие теги. ⚠️ Привязка к кампании
  происходит в момент УТВЕРЖДЕНИЯ — перед закрытием акции очередь валидации разобрать до нуля.
- Группы АЗС (геймификация, лидерборды), инвайт-коды, `/report` (HTML/ZIP) и `/xlsx` за период,
  `/stats` с бейджем pending, ежедневный scheduler (неактивность АЗС, зависшие pending >7 дней)

## Стек

Python + aiogram 3 + SQLite. Деплой на Linux-хостинг (есть `Dockerfile`, `k8s/`, `install.sh`, `update.sh`, systemd-юнит).

**Деплой:** k8s-ready, `/data` PVC, `healthz` endpoint, логи в stdout. См. `receipts_bot/HANDOFF.md`.

## Принципы

- Кассирам нужен максимально простой UX: кнопка акции + фото. Единственное разрешённое
  исключение — вопрос «сколько акционных товаров?» после комбо-фото (решение Мадияра,
  2026-07-03): ответ кнопкой 1–5+ или цифрой текстом, необязательный.
- Вся аналитика и отчётность — на стороне бота и владельца.
- Новые функции по умолчанию — **админские**; кассирский флоу расширять только по явному запросу.
- Race-sensitive записи в БД — через `_connect_write()` (BEGIN IMMEDIATE), не через `_connect()`.

## Не делай

- Не усложняй кассирский флоу без явного запроса (см. Принципы).
- Не коммить изменения по боту в Marketing-Director — только в свой репо `cheki-bot`.
- Не публикуй `.env` или содержимое `receipts/`.
- Не читай `receipts.promotion`/`campaign_id` (legacy) для агрегаций — только junction-таблицы.
